2013年6月7号晚数据中新突然报告说ESB数据库有一个节点宕机了 数据库版本ORACLE 11.2.0.3
我去协助分析问题，发现有一个169.254网段的IP地址PING不通，由于的ORACLE 11G版本中有一个新功能，允许用户选用多块物理网卡作为RAC的心跳来用，这个功能会自动生成一个169.254.xx.xx的虚拟IP，如果这个IP不通会引起RAC节点宕。
使用一下命令看路由信息
$ route
Kernel IP routing table
Destination Gateway Genmask Flags Metric Ref Use Iface
......
169.254.0.0 * 255.255.0.0 U 0 0 0 bondib0
......
发现这条路由不见了。手动加上这条路由网络就能够恢复正常。
之后又看了/var/log/messages日志，发现有如下日志：

从MOS的文档里查到这个问题就是IP有冲突。
IBM的PC服务器有个使用USB做为管理网络的特性（IBM独有，别的品牌服务器中没有发现）。在你没有连接USB网卡的时候会不停向DHCP申请IP，如果没有发现DHCP时就会默认分配一个169.254.xx.xx的IP地址，这个时候就会和ORACLE的虚拟IP产生冲突。造成路由信息丢失。从网上看这个USB0的问题还会引起LINUX HA异常。
解决方法：
#ifdown usb0
vi /etc/sysconfig/network-scripts/ifcfg-usb0
onboot=no
关闭usb0网卡功能（现在还没听说过有人用过这个功能的。如果用的话只有IP地址不是169.254.xx.xx也一样不会出现这个问题）。
关闭后重启停止的crs后问题解决。
